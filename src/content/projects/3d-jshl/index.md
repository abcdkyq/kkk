## 1. 项目背景与立项动机

在传统跑酷赛车游戏中，如果采用静态关卡或有限地图，往往会出现如下问题：

- **重复感强**：玩家在较短时间内就能记住地图结构，刺激性下降；
- **资源压力大**：大地图一次性加载会占用较多内存，影响低端设备体验；
- **地形过渡不自然**：随机地形拼接如果没有边界平滑处理，容易出现视觉/碰撞断层；
- **玩法循环单一**：缺少动态障碍与即时反馈，玩家留存下降。

本项目围绕“**小体量、高复玩、可拓展**”目标，构建了一款在**程序化圆柱赛道**上进行的无限跑酷赛车游戏。其核心价值在于：

1. 通过程序化生成实现低资源成本下的高变化内容；
2. 通过世界移动架构实现无限前进体验；
3. 通过物理组件 + 特效反馈增强驾驶手感；
4. 通过得分门与障碍形成“风险—收益”决策节奏。

## 2. 项目目标与范围定义

### 2.1 目标定义

本项目旨在实现一个可运行、可迭代、可展示工程能力的完整玩法 Demo，并覆盖以下技术目标：

- 完成圆柱赛道的程序化网格生成；
- 支持连续地形块生成与平滑衔接；
- 构建无限跑酷循环（地图滚动、事件触发、结算重开）；
- 打通输入、物理、特效、UI、音频、动画全链路。

### 2.2 范围边界

**已包含：**

- 程序化地形（圆柱形 + 噪声起伏）
- 双区块流式更新
- 障碍/得分门生成与交互
- 车辆转向与基础物理反馈
- 草地粒子与轮胎印
- 主界面与结算界面
- 分数与历史最佳记录

**暂未包含：**

- 多车型系统
- 复杂 AI 对手
- 联机同步
- 高级地形材质混合（如多层贴图地貌系统）
- 完整关卡编辑器

## 3. 用户体验与玩法设计

### 3.1 核心体验目标

- **直觉控制**：鼠标点击/键盘方向即可控制转向，低学习门槛；
- **持续刺激**：前方地形与障碍持续变化，避免机械重复；
- **即时反馈**：碰撞失败、得分提示、时间累计、轮胎痕迹等都应实时可感；
- **失败可重试**：快速重开，形成短回合高复玩循环。

### 3.2 玩法主循环（Game Loop）

1. 玩家进入主菜单并开始游戏；
2. 车辆在圆柱赛道上进行转向控制；
3. 躲避障碍、穿越得分门提升分数；
4. 被障碍命中或触发失败条件后进入结算；
5. 显示本局分与历史最佳，支持一键重开。

该循环具备“即时上手 + 高频反馈 + 快速重试”的典型街机跑酷结构。

## 4. 系统需求分析

### 4.1 功能需求

1. 支持按参数生成圆柱网格（环向分段、纵向分段、世界缩放）；
2. 支持噪声地形起伏并可调随机性；
3. 支持地形块的无缝拼接；
4. 支持障碍物与得分门随机生成；
5. 支持玩家控制、碰撞检测与失败判定；
6. 支持分数计数、计时显示、最佳分持久化；
7. 支持主界面与结算动画。

### 4.2 非功能需求

1. **性能稳定**：长时间运行不卡顿、不明显内存膨胀；
2. **鲁棒性**：不因某单次生成异常导致流程中断；
3. **可调参性**：核心难度参数可在 Inspector 快速调节；
4. **可扩展性**：后续可接入新障碍、新地形规则、新计分模式。

## 5. 总体架构设计

项目采用分层职责设计：

- **世界层**：赛道网格生成、块更新、世界移动；
- **玩法层**：障碍/门生成、触发逻辑、失败判定；
- **角色层**：车辆控制、轮子表现、特效反馈；
- **状态层**：分数、计时、结算、重开；
- **表现层**：相机跟随、UI 动画、音频。

这种结构有两个直接收益：

1. 模块边界清晰，便于单独调试与替换；
2. 玩法扩展时，影响范围可控（例如仅替换生成策略，不影响 UI 管线）。

## 6. 核心模块详解

### 6.1 世界生成系统（WorldGenerator）

**职责：**

- 动态创建圆柱赛道 Mesh；
- 基于噪声塑造地形起伏；
- 在赛道表面随机生成障碍和得分门；
- 管理双区块更新，实现无限赛道。

**关键能力：**

1. `CreateCylinder()`：创建地形块 GameObject，挂载 MeshFilter、MeshRenderer、MeshCollider；
2. `Generate()`：产出可渲染网格；
3. `CreateShape()`：计算顶点、UV、三角形；
4. `UpdateWorldPieces()`：滚动更新地形块；
5. `CreateItem()`：按概率投放 Gate 或 Obstacle。

**参数化思路：**

- `dimensions` 控制网格密度（横向环分段、纵向长度分段）；
- `pointScale/worldScale` 控制赛道尺寸；
- `perlinScale/waveHeight/offset/randomness` 控制地形风格与变化；
- `startObstacleChance/gateChance` 控制玩法风险收益比。

### 6.2 世界移动系统（BasicMovement）

**职责：**

- 让地形块按固定速度前移（模拟车辆向前）；
- 根据车辆转向量同步旋转世界，构建“玩家静止、世界转动”的相对运动效果。

**意义：**

- 降低玩家对象高速度运动带来的浮点/碰撞不稳定风险；
- 将“无限前进”问题转化为“有限对象循环位移”问题，便于优化。

### 6.3 车辆控制与物理表现（Car）

**职责：**

- 读取输入并更新目标转向；
- 使用 WheelCollider 同步车轮视觉；
- 基于 Raycast 判断接地并驱动草地特效；
- 检测转向差触发轮胎印；
- 失败后生成 ragdoll 并隐藏原车体。

**交互设计细节：**

- 支持鼠标与键盘双输入路径；
- 未输入时自动回正目标角；
- 接地与非接地状态采用不同处理（非接地时施加后部下压力提升稳定性）。

### 6.4 游戏状态与结算管理（GameManager）

**职责：**

- 维护分数、计时器、失败状态；
- 更新 HUD；
- 处理失败逻辑（动画、音频、停止世界运动、显示结算）；
- 持久化最佳分（PlayerPrefs）。

**特点：**

- 对外仅暴露必要接口（如 `UpdateScore`、`GameOver`）；
- 通过统一管理器确保结算流程一致，避免重复触发。

### 6.5 玩法实体（Gate / Obstacle / CarGameOverTrigger）

- **Gate**：触发得分并播放音效，内部包含防重复加分保护位；
- **Obstacle**：与玩家碰撞后触发游戏失败；
- **CarGameOverTrigger**：用于辅助触发失败条件，提升边界场景鲁棒性。

### 6.6 相机与表现系统（CameraFollow）

- 采用阻尼跟随与高度插值，保证运动画面平滑；
- 开局阶段使用较低阻尼，进入交互后再切换为正常阻尼，提升观感；
- 始终 LookAt 玩家目标，强化速度感与方向感。

### 6.7 菜单与音频（MainMenu / Music）

- `MainMenu`：处理开始输入、播放 UI 动画并切场景；
- `Music`：单例 + `DontDestroyOnLoad` 实现跨场景背景音乐持续播放。

## 7. 程序化圆柱赛道生成算法

本项目的地形算法不是使用现成 primitive，而是从网格底层数据手工构建：

1. **顶点生成**：
   - 设横向分段 `xCount`、纵向分段 `zCount`；
   - 使用角度映射 `angle = x * 2π / xCount`；
   - 将环向点投射到圆柱截面：
     - `x = cos(angle) * radius`
     - `y = sin(angle) * radius`
     - `z = z * pointScale`

2. **UV 计算**：
   - 使用 `x,z` 对应纹理坐标，保证贴图连续性；
   - 可进一步扩展为按弧长归一化 UV，减少拉伸。

3. **噪声形变**：
   - 使用 Perlin Noise 采样，得到每个顶点扰动量；
   - 沿“朝向圆柱中心方向”位移，形成赛道起伏。

4. **三角索引构建**：
   - 每个四边形网格单元拆成两个三角形；
   - 按固定 winding 顺序保证法线一致。

5. **法线重计算**：
   
   - `mesh.RecalculateNormals()` 提供正确光照响应。

该实现体现了对 Unity Mesh 数据结构和程序化建模流程的掌握。

## 8. 无缝拼接与平滑过渡策略

在随机地形中，最常见问题是“块间接缝台阶”。本项目采用以下策略：

1. 在上一块末尾保存一圈顶点（`beginPoints数组`）；
2. 下一块前 `startTransitionLength` 行顶点不直接使用噪声值，而是与上一块末尾行顶点（beginPoints数组）进行线性插值：

最终顶点 = t × 新噪声顶点 + (1-t) × 上一块末尾顶点

其中t 从 0 递增到 1，即从完全沿用上一块 → 完全用新噪声，中间线性过渡。
补充：t 的计算方式：t = z / startTransitionLength，其中 z 是当前行在新块中的纵向索引；
边界条件：t=0 时顶点完全等于上一块末尾点（无缝），t=1 时完使用 Perlin 噪声值

**效果：**

- 边界连续；
- 视觉“突变感”显著降低；
- 车辆通过接缝时物理反馈更稳定。

## 9. 无限跑酷实现机制（双区块滚动）

### 9.1 机制描述

系统仅维护两个世界块：`pieces[0]` 与 `pieces[1]`。

- 初始化时生成两块，首尾相接；
- 当前方块移动到阈值后：
  1. 销毁旧后块；
  2. 前块成为新的后块；
  3. 生成新前块并对齐到后块末端；
  4. 对新前块执行地形更新与内容投放。

### 9.2 优点

- 常量级场景对象数量；
- 无限时长运行不会因地图长度线性增长；
- 更易控制内存占用与碰撞体数量。

### 9.3 可进一步优化

- 用对象池替代销毁/实例化，减少 GC 和 CPU 峰值；
- 将生成分摊到多帧，避免单帧突刺。

## 10. 交互、物理与反馈设计

### 10.1 输入层

- 鼠标点击：屏幕左右分区控制左右偏转；
- 键盘轴输入：按比例映射目标转向角；
- 无输入时回正，降低操控负担。

### 10.2 车辆与地形接触

- 使用 WheelCollider + 轮子可视模型分离策略；
- 每帧通过 `GetWorldPose` 维持视觉与碰撞一致；
- 通过 Raycast 判断后轮接地状态。

### 10.3 反馈系统

- **草地粒子**：接地时激活并贴合轮胎高度；
- **轮胎印**：检测转向变化并定时在后轮位置生成；
- **失败反馈**：碰撞后触发音效 + 动画 + 车体解体。

该反馈链路让玩家能在视觉上“读懂”车辆状态，提升可控感与沉浸感。

## 12. 性能分析与优化思路

### 12.1 当前性能控制手段

1. 双区块常量场景规模；
2. 世界移动代替玩家高速长位移；
3. 仅在需要时启停特效对象；
4. 关键逻辑集中于少数核心脚本。

### 12.2 潜在性能热点

- 频繁 `Instantiate/Destroy`（地形块与轮胎印）；
- MeshCollider 生成开销；
- `FindObjectOfType` 在初始化时的查找成本；
- 高分辨率网格与高密度障碍导致碰撞压力上升。

### 12.3 进阶优化建议

1. **对象池化**：赛道块、障碍、门、轮胎印统一池化；
2. **组件缓存**：减少运行期对象查找；
3. **层级碰撞裁剪**：合理设置 Layer Matrix；
4. **LOD / 简化碰撞网格**：降低远端成本；
5. **参数分档**：按设备等级动态调节网格密度与刷新率。

## 13. 稳定性与可维护性分析

### 13.1 稳定性

### 13.3 可测试性

- 可围绕“生成正确性”“过渡连续性”“碰撞触发可靠性”“结算一致性”建立测试矩阵；
- 可通过固定随机种子进行可复现调试。

## 14. 项目难点与技术亮点总结

## 难点

1. **圆柱地形的程序化网格实现复杂**：需要同时保障几何正确、索引正确与 UV 连续；
2. **随机地形的拼接连续性**：需处理块边界过渡，避免视觉与碰撞断层；
3. **无限场景与性能权衡**：既要无限，又要稳定；
4. **物理表现与视觉反馈一致性**：轮子、粒子、轮胎印、碰撞必须联动。

## 亮点

1. **手写 Mesh 数据流**（顶点/UV/三角）+ 噪声地形；
2. **边界插值平滑拼接策略**，明显提升连续体验；
3. **世界移动 + 双区块复用**，实现轻量无限跑酷；
4. **完整玩法闭环**（输入—驾驶—得分—失败—重开）；
5. **多系统协同能力**（物理/特效/UI/动画/音频一体化）。

## 15. 测试方案与质量保障

### 15.1 功能测试

- 地形是否按参数生成；
- 障碍与门是否按概率分布生成；
- 穿门得分是否只触发一次；
- 障碍碰撞是否稳定触发失败；

### 15.2 体验测试

- 转向是否灵敏、回正是否自然；
- 相机跟随是否晕动；
- 地形接缝处是否突兀；
- 长时间游玩是否出现明显卡顿。

### 15.3 回归测试

- 修改地形参数后，验证碰撞与通过性；
- 修改障碍概率后，验证难度变化是否符合预期；
- 修改 UI 动画后，验证场景切换与输入响应。

## 16. 项目复盘与经验沉淀

1. **先稳定核心循环，再打磨表现层**：先确保“能玩且稳定”，再追求“好看”；
2. **程序化系统必须可视化调参**：参数暴露能显著提高迭代效率；
3. **无限玩法本质是资源管理问题**：不是“无限生成”，而是“有限复用”；
4. **反馈决定手感上限**：粒子、音效、镜头阻尼对驾驶体验影响极大；
5. **模块边界清晰可显著降低维护成本**。
